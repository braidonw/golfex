# Golfex: Golf Club Booking App — Design Spec

Rewrite of the Rust `axum-booker` application into Elixir/Phoenix. Multi-user, multi-club golf booking automation tool that integrates with MiClub-powered golf club websites.

## Overview

Users log in, add golf clubs (with their MiClub credentials), browse events, view tee time details, and book slots — either immediately or via scheduled jobs that execute at a specified time.

## Architecture

Phoenix monolith with well-separated contexts. LiveView for all interactive pages. PostgreSQL via Ecto. Oban for scheduled bookings. Sugarcube/CUBE CSS for styling.

The project uses `binary_id: true` — all primary keys are UUIDs.

## Contexts & Modules

### Golfex.Accounts

Generated by `phx.gen.auth` (already scaffolded) using Phoenix 1.8's scoped auth. The project uses `Golfex.Accounts.Scope` for threading user context through LiveViews — the current scope is available as `@current_scope` in LiveView assigns (configured via `assign_key: :current_scope` in config).

Email/password authentication with session management. No public registration — the existing `/users/register` route should be removed from the router. All context functions that touch user-scoped data take a `%Scope{}` struct.

All existing auth routes and controllers (`/users/log-in`, `/users/log-out`, `/users/settings`, `/users/update-password`) are already generated and should be kept as-is.

### Golfex.Clubs

CRUD for clubs and user-club associations. Each user-club record stores encrypted MiClub credentials (member ID, username, password) using Cloak.Ecto.

Cloak setup requires:
- A `Golfex.Vault` module configured with AES-GCM cipher
- Vault added to the application supervision tree
- `CLOAK_KEY` environment variable for the encryption key

Public API:
- `list_clubs_for_user/1`
- `get_user_club!/2`
- `add_club_for_user/2`
- `update_user_club/2`
- `remove_club_for_user/2`

### Golfex.MiClub

HTTP client for interacting with MiClub-powered golf club websites. Uses Req with cookie jar for session management.

MiClub endpoints return mixed formats:
- Events list → JSON (parsed with Jason)
- Event detail → XML (parsed with SweetXml)
- Booking → XML (parsed with SweetXml)

Submodules:
- `Client` — HTTP requests with cookie-based session management. Authenticates once per user action (e.g. viewing events page), reuses the session within that action, discards after. No long-lived sessions stored in DB.
- `Auth` — login to MiClub, handle session expiry by re-authenticating
- `Parser` — response parsing into Elixir structs (JSON and XML)

Public API:
- `login/1` — authenticate with a club (takes a `UserClub` struct)
- `list_events/1` — fetch all events (takes a `UserClub` struct)
- `get_event/2` — fetch event detail with booking sections (takes `UserClub`, event_id)
- `get_booking_group/3` — fetch booking group detail (takes `UserClub`, event_id, group_id)
- `book/4` — execute a booking (takes `UserClub`, booking_group_id, row_id, member_id)

The `book/4` function posts to MiClub's Ajax booking endpoint with `row_id` (the specific booking entry slot), `member_id`, `myGroup` (group ID), and `findAlternative` parameters.

Error handling: MiClub HTTP interactions should return `{:ok, result}` or `{:error, reason}` tuples. Handle timeouts (default 30s), network errors, authentication failures (re-auth and retry once), and unexpected response formats. Invalid credentials should surface a clear error to the user rather than silently retrying.

### Golfex.Events

Manages cached events fetched from MiClub.

Cache strategy:
- Events are fetched on demand when a user views a club's events page
- If cached data is older than 12 hours, fetch fresh from MiClub and upsert
- If cached data is less than 12 hours old, serve from database
- Manual "Refresh" button forces a fresh fetch regardless of cache age
- A successful booking invalidates the cached event (availability changes)
- Booking group/section detail is always fetched live (too granular and time-sensitive to cache)

Public API:
- `list_events/1` — read from cache (triggers sync if stale)
- `get_event!/2` — read single event from cache
- `sync_events/1` — fetch from MiClub and upsert into local cache
- `invalidate_event/1` — mark a cached event as stale after booking

### Golfex.Bookings

Manages scheduled bookings via Oban.

Oban configuration:
- Queue: `bookings` with concurrency of 1 (serial execution to avoid race conditions)
- Added to supervision tree in `application.ex`

Two booking modes:
1. **Book Now** — immediately execute booking via MiClub client
2. **Schedule Booking** — create a scheduled_booking record + enqueue an Oban job for a specified datetime

`BookingWorker` (Oban worker) executes the booking at the scheduled time. On failure, Oban handles retries (default 3 attempts). Updates scheduled_booking status to completed/failed.

Scheduled booking status state machine:
- `pending` → initial state when user schedules a booking
- `running` → Oban worker has picked up the job and is executing
- `completed` → booking succeeded
- `failed` → booking failed after all retry attempts exhausted
- `cancelled` → user cancelled the booking before execution

Public API:
- `book_now/4` — immediate booking (takes `UserClub`, booking_group_id, row_id, member_id)
- `schedule_booking/1` — create scheduled booking + Oban job (takes a map with: user_id, user_club_id, event_id, miclub_event_id, miclub_group_id, miclub_row_id, miclub_member_id, scheduled_for)
- `cancel_booking/1` — cancel pending booking (cancels Oban job, marks record cancelled)
- `list_bookings_for_user/1` — user's booking history

## Styling

### Sugarcube + CUBE CSS

Replaces Tailwind. Uses design tokens (CSS custom properties), global styles, compositions (layout primitives), blocks (component-specific), and utilities.

The sugarcube files are currently generated under `assets/js/` and need to be moved to `assets/css/`. Target file structure:

```
assets/
├── css/
│   ├── app.css              — main entry point, imports everything
│   ├── design-tokens/       — JSON token files (from sugarcube generate)
│   ├── styles/
│   │   ├── global/          — resets, base typography
│   │   ├── compositions/    — flow, cluster, grid, sidebar, etc.
│   │   ├── blocks/          — component-specific (prose, etc.)
│   │   └── utilities/       — region, visually-hidden, etc.
│   └── components/ui/       — sugarcube component CSS (button, card, input, etc.)
├── js/
│   └── app.js               — JS only, no CSS
```

Build pipeline: `sugarcube generate` CLI produces token CSS files. Add a second esbuild config entry for CSS bundling (esbuild supports CSS natively). The CSS entry point is `css/app.css`, outputting to `priv/static/assets/css/app.css`. Add a corresponding watcher in `dev.exs` for CSS rebuilds.

Templates use semantic classes and CUBE CSS compositions rather than utility classes.

## Deployment

Standard Phoenix release Dockerfile (multi-stage: build → release). PostgreSQL as a separate service on Coolify.

Environment variables:
- `DATABASE_URL` — PostgreSQL connection string
- `SECRET_KEY_BASE` — Phoenix secret
- `CLOAK_KEY` — encryption key for MiClub credentials (base64-encoded 32-byte key)
- `PHX_HOST` — production hostname
- `PORT` — HTTP port

## Testing

- MiClub HTTP interactions: use `Req.Test` with plug-based mocks for deterministic testing
- Oban jobs: use `Oban.Testing` for inline job execution in tests
- LiveView pages: standard Phoenix.LiveViewTest
- Context modules: standard Ecto sandbox tests
